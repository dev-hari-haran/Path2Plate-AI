📋 Project Overview

Food Delivery Bot is a real-time autonomous navigation simulator built entirely in the browser. It simulates a delivery robot navigating a 20×20 grid from a start position to a burger goal (🍔), avoiding obstacles and collecting sub-goals along the way.

The simulation uses the D* Lite incremental pathfinding algorithm, which replans routes on-the-fly when new obstacles appear — without restarting from scratch. The project also includes a full ESP8266 hardware bridge that lets you control a real physical robot using WebSocket, HTTP, or MQTT.

ℹ️
This project runs entirely in a single HTML+CSS+JS file. No build system or server required. Open index.html in any modern browser.

🚀 Quick Start

  1. Open index.html in your browser (Chrome/Firefox/Edge).
  2. The grid initialises with bot at (1,1) and goal 🍔 at (18,18).
  3. Use Map Editor buttons to place obstacles, subgoals, or change start/goal.
  4. Select Manual or Auto mode in the Drive Mode panel.
  5. In Manual: use the D-pad or keyboard arrow keys to move the bot.
  6. In Auto: press ▶ START NAV to begin autonomous navigation.
  7. When the bot reaches 🍔 the delivery complete window appears and the violet path is painted.
⚠️
SubGoal tip: Add subgoals before starting navigation. The SubGoal button is automatically disabled while the simulation runs and re-enabled on stop or mission complete.

🧠 D* Lite Pathfinding

D* Lite (Dynamic A* Lite) is an incremental heuristic search algorithm designed for real-time replanning. Unlike plain A*, it only updates the portion of the search graph that changed when a new obstacle is detected — making it extremely efficient for dynamic environments.

How This Implementation Works

The sim uses an A* replanner that runs a full A* search from the current bot position through all subgoals to the goal. This is triggered on every:

  • New obstacle placed on grid
  • Moving obstacle that shifts into a new cell
  • ESP8266 sensor packet received (obstacle in front of bot)
  • Mode switch from Manual → Auto
  • SubGoal collected (replans remaining route)

🎮 Drive Modes

The Drive Mode panel sits parallel to the grid on the left side, always visible. Toggle between Manual and Auto without scrolling.

Manual Mode

Control the bot with the D-pad or keyboard. The dashed cyan line shows the optimal shortest path to the goal. The path panel shows:

  • OPTIMAL — Shortest possible steps remaining
  • YOUR STEPS — Steps you have taken
  • EXTRA — How many steps you wasted
  • EFFICIENCY — % score (100% = perfect)

Auto Mode

The bot autonomously follows the D*-computed path. Press ▶ START NAV to begin. Before running, the simulation will:

  1. Check all subgoal segment distances — any > 10 cells triggers priority dialog.
  2. Sort remaining subgoals using nearest-neighbour algorithm.
  3. Compute full path and start traversal.

Speed Control

The speed slider sets traversal rate from 1 cell/sec to 10 cells/sec. Can be adjusted while running.

🔄 Moving Obstacles

Click 🔄 Moving Obstacles in the Map Editor to place 5 dynamic obstacles. Each one is randomly assigned either horizontal (left ↔ right) or vertical (top ↔ bottom) movement within a bounded lane.

Behaviour

PropertyValue
Step intervalEvery 600ms
Movement axesHorizontal (H) or Vertical (V) — random per obstacle
BounceReverses direction at lane boundaries or blocked cells
Collision avoidanceNever enters bot, start, goal, or subgoal cells
D* replan triggerIf bot's next step is blocked by moving obstacle, instant replan

Visual

Moving obstacles show an orange pulsing ring and a directional arrow (→ ← ↑ ↓) so you can see their movement direction at a glance.

⚠️
Moving obstacles are cleared when you press ↺ Clear All. Toggle them off with the same button (it switches between "Moving Obstacles" and "Stop Moving Obs").

📍 SubGoals & Priority System

SubGoals are intermediate waypoints the bot must visit before reaching the final goal. They appear on the grid as blue S1, S2, … cells (high-priority subgoals are shown in orange with a ★ star).

Placing SubGoals

  1. Select 🔵 SubGoal in the Map Editor.
  2. Click any free cell on the grid. SubGoal S1, S2, … are added in order.
  3. The button is automatically disabled when simulation starts and re-enabled on stop.

Auto-Sort on Start Nav

When you press ▶ START NAV, subgoals are automatically reordered by a nearest-neighbour TSP approximation — starting from the bot's current position, always picking the closest remaining subgoal next. This avoids inefficient zig-zag routes.

Priority Check Dialog

Before navigation starts, if any subgoal-to-subgoal segment (or start→subgoal) requires more than 10 A* steps, a priority dialog appears for that subgoal:

OptionEffect
★ High Priority — IncludeMarks the subgoal as high priority (orange ★). Included regardless of distance.
✕ Skip SubGoalRemoves the subgoal from the route entirely.
▶ Keep NormalKeeps the subgoal but at normal priority — included in route.
ℹ️
The dialog cycles through all "too far" subgoals one at a time. After resolving each, the nearest-neighbour sort is applied and navigation begins.

🕹 Manual Mode Guidance

Shortest Path Overlay

In Manual mode, a dashed cyan line constantly shows the optimal route from the bot's current position to the goal (via subgoals). This recalculates after every move.

Deviation Warnings

If you move in a direction that is not the next step on the optimal path, the System Log shows a warning:

⚠ Deviation! Optimal next: (5,3) — you went (4,3). Optimal remaining: 12 steps

Warnings are rate-limited to every 3 steps to avoid log spam.

Orange Trail

Every cell the bot visits is marked in orange as a breadcrumb trail, showing where you've been.

Violet Completed Path

The bot's actual route is permanently painted in violet/purple with a glowing line, dot waypoints, and a "✓ DELIVERED" label at the midpoint. It stays visible until you press Clear All.

⚡ ESP8266 Hardware Bridge

The sim can control a real ESP8266-based robot over your local WiFi network. Three communication protocols are supported.

WebSocket (Recommended)

Bi-directional, low-latency. The ESP runs a WebSocket server and the browser connects as a client.

#include <ESP8266WiFi.h> #include <WebSocketsServer.h> WebSocketsServer webSocket = WebSocketsServer(80); // Receive: {"cmd":"FORWARD"} | "BACKWARD" | "LEFT" | "RIGHT" | "STOP" // Send: {"type":"obstacle","dist":12.5,"unit":"cm"}
ℹ️
When the ESP sends an obstacle packet, the browser immediately places an obstacle in the cell in front of the bot and triggers a D* replan.

HTTP REST (Polling)

The browser polls GET /status every 500ms and sends commands via POST /cmd.

// Command endpoint POST http://<esp-ip>/cmd Body: {"cmd":"FORWARD"} // Status endpoint GET http://<esp-ip>/status Returns: {"obstacle":true,"dist":14.2}

MQTT (Wireless Broker)

Uses broker.hivemq.com (public broker). Include mqtt.js in your page.

// Subscribe: foodbot/obstacle → receive obstacle events // Publish: foodbot/cmd → send motor commands

Sensor Setup

The ESP uses an HC-SR04 ultrasonic sensor (Trig: D6, Echo: D7) that scans every 300ms. If an object is detected closer than 20cm, it sends an obstacle packet.

CSS Variables System

Every colour is a CSS custom property on :root / [data-theme="dark"] / [data-theme="light"]. Canvas drawing reads these at runtime via:

function cv(name) { return getComputedStyle(document.documentElement).getPropertyValue(name).trim(); } // Usage: ctx.fillStyle = cv('--trail-color');

This allows instant theme switching with zero canvas code changes.
